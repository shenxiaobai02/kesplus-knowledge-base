# 角色化提示词模板集

## 用于 AI Agent 底座项目全生命周期提效

本提示词集专为 **AI Agent 底座开发项目** 设计，覆盖产品经理、架构师、开发工程师、测试工程师四大角色在项目各阶段的核心任务。每个提示词都经过结构化设计，可直接复制使用（替换 `{占位符}` 为具体内容），帮助团队高效利用 LLM 完成需求分析、架构设计、代码生成、测试用例编写等关键工作。

---

## 一、产品经理（Product Manager）

### 1.1 需求分析与用户故事生成

```markdown
# 角色
你是一位资深的 AI 产品经理，专注于企业级智能体平台。

# 任务
基于以下业务背景，生成完整的用户故事地图和功能需求列表。

# 业务背景
{粘贴业务描述：如“构建一个客服智能体，支持多轮对话、自动查询订单、转人工”}

# 输出要求
1. 按照“史诗 → 功能 → 用户故事”三层结构输出。
2. 每个用户故事遵循格式：“作为 {角色}，我想要 {功能}，以便 {价值}”。
3. 标注优先级（P0/P1/P2）和预估工作量（人天）。
4. 识别非功能性需求（并发量、响应时间、安全性）。

# 输出格式
Markdown 表格，包含：需求ID、类型、描述、优先级、工作量、验收标准。
```

### 1.2 竞品分析与功能定位

```markdown
# 角色
AI 产品战略专家。

# 任务
对以下竞争对手的智能体产品进行对比分析，并给出我们的差异化定位建议。

# 竞品列表
- LangChain4j
- Spring AI Alibaba  
- AgentScope-Java
- 其他：{列出}

# 分析维度
1. 模型集成能力
2. 多智能体协作
3. 企业级可观测性
4. 开发体验与学习曲线

# 输出要求
- SWOT 分析矩阵
- 我们的核心差异化功能（至少3个）
- 建议的 MVP 功能范围
```

### 1.3 产品发布说明生成

```markdown
# 角色
技术产品经理。

# 任务
根据以下 Git 提交记录和 Issue 列表，生成面向开发者的 Release Notes。

# 输入
- 版本号：{v1.2.0}
- 提交记录：{粘贴 git log --oneline 输出}
- 关闭的 Issue：{列表}

# 输出要求
1. 分类组织：新特性 / 改进 / 修复 / 废弃 / 安全更新
2. 每个条目附带 Issue/PR 链接（如有）
3. 包含升级指南（破坏性变更、配置迁移步骤）
4. 致谢贡献者

# 格式
Markdown，适合发布在 GitHub Releases 页面。
```

## 二、架构师（Architect）

### 2.1 技术选型与对比分析

```markdown
# 角色
企业级软件架构师，拥有 15 年以上 Java 生态经验。

# 任务
对以下技术组件进行选型评估，输出对比表和最终推荐。

# 场景描述
{我们需要构建一个 AI Agent 底座，支持 1000 QPS，多租户，需要长期记忆和工具调用}

# 候选技术
- 模型集成：LangChain4j vs Spring AI vs 自研
- 向量数据库：Milvus vs Qdrant vs PGVector
- 编排引擎：LangGraph4j vs AgentScope vs 自研状态机

# 评估维度（每项1-5分）
- 性能、社区活跃度、学习曲线、与现有栈匹配度、生产案例、许可协议

# 输出
1. 评分对比表
2. 最终推荐组合及理由
3. 备选方案与风险规避策略
```

### 2.2 系统架构设计评审

```markdown
# 角色
解决方案架构师。

# 任务
基于以下需求，设计微服务架构图、数据流图，并指出可能的技术债务。

# 需求
{粘贴架构师收集的需求：如“需要支持 Workflow 和 Agentic 双引擎，会话外置……”}

# 输出要求
1. 架构分层描述（Mermaid 代码）
2. 关键组件职责及交互接口
3. 数据流向图（包括控制流和数据流）
4. 风险点列表（单点故障、性能瓶颈、安全暴露面）
5. 扩展性建议（如何新增模型提供商、工具类型）

# 额外约束
遵循云原生原则，可部署在 Kubernetes。
```

### 2.3 技术债务与重构计划

```markdown
# 角色
技术架构治理负责人。

# 任务
分析以下代码仓库的架构质量问题，制定重构路线图。

# 输入
- 代码仓库摘要：{粘贴目录结构或关键类列表}
- 已知痛点：{例如：“模型适配器耦合严重”、“记忆模块与业务逻辑混在一起”}

# 输出要求
1. 按优先级（高/中/低）列出技术债务项
2. 每个债务项的影响范围、估计重构工作量（人天）
3. 分阶段的重构计划（每阶段目标、持续时间、验收指标）
4. 如何避免未来产生类似债务的设计原则
```

## 三、开发工程师（Developer）

### 3.1 代码生成与实现（按任务）

```markdown
# 角色
资深 Java 后端开发工程师，熟悉 Spring Boot 3、LangChain4j、Reactive 编程。

# 任务
实现一个 {组件名称}，需求如下：

{粘贴详细功能规格或用户故事}

# 技术栈约束
- Java 17
- Spring Boot 3.2.x
- LangChain4j 1.15.1
- Maven 多模块

# 输出要求
1. 完整的 Java 类代码，包含必要导入、注解、日志。
2. 单元测试（JUnit 5 + Mockito）。
3. 必要的配置项（application.yml 片段）。
4. 代码注释解释关键逻辑。

# 示例输入输出
{提供期望的 API 请求/响应示例}
```

### 3.2 接口设计与 OpenAPI 规范

```markdown
# 角色
后端开发工程师，擅长 RESTful API 设计与 OpenAPI 3.0。

# 任务
设计一个 {智能体执行} 的 REST API，并提供 OpenAPI YAML 定义。

# 需求
- 路径：/api/v1/agents/{agentId}/execute
- 方法：POST
- 请求体：{用户消息，可选参数（stream=false, sessionId）}
- 响应：SSE 流（当 stream=true）或 JSON（当 stream=false）

# 输出要求
1. OpenAPI 3.0 YAML 内容
2. 请求/响应 JSON 示例
3. 错误码定义（400, 429, 500 等）
4. 鉴权方式说明（Bearer Token）

# 附加要求
支持根据 Accept 头自动选择返回格式。
```

### 3.3 重构与优化（性能/可读性）

```markdown
# 角色
Java 性能优化专家。

# 任务
重构以下代码片段，提升执行效率，同时保持功能一致。

# 待重构代码
```java
{粘贴现有代码，例如一个循环调用 LLM 或多次查询数据库的方法}
```

# 优化目标
- 减少方法总耗时（当前平均 2.3 秒）
- 降低内存分配频率
- 提高代码可读性

# 输出
1. 重构后的代码
2. 优化点说明（例如：批量调用、并行流、缓存）
3. 预期性能提升（百分比或具体数值）
4. 可能的副作用（如最终一致性、额外的缓存失效逻辑）
```

## 四、测试工程师（Test Engineer）

### 4.1 测试用例生成（覆盖正向/负向）

```markdown
# 角色
AI 测试工程师，擅长黑盒、白盒及 LLM 输出质量评估。

# 任务
为以下功能模块编写完整的测试用例，包括功能、异常、边界条件。

# 被测模块
{描述：如“智能体工具调用函数，输入 SQL 查询语句，返回查询结果”}

# 输入约束
- 支持 SELECT 语句，不允许 DDL/DML
- 超时 5 秒
- 结果行数限制最多 100 行

# 输出要求
Markdown 表格，列包括：
- 用例ID
- 测试场景（描述）
- 前置条件
- 输入数据
- 预期输出（包括正常返回、错误码、异常消息）
- 自动化级别（高/中/低）

# 额外要求
包含至少 3 个 LLM 输出质量相关的测试（例如：幻觉检测、格式约束）。
```

### 4.2 性能测试脚本（JMeter/Gatling）

```markdown
# 角色
性能测试工程师，熟练使用 Gatling。

# 任务
基于以下 API 规格，生成 Gatling 压力测试脚本。

# API 规格
- 端点：POST /api/v1/agents/order-assistant/execute
- 请求体：{"message": "帮我查询订单 O12345", "sessionId": "user-${random}"}
- 响应：JSON 包含 answer 字段

# 测试负载
- 并发用户：50 -> 200 爬升，持续 10 分钟
- 思考时间：均匀分布 1-3 秒
- 断言：状态码 200，响应时间 < 2 秒（P95）

# 输出要求
1. Gatling 仿真类 Scala 代码
2. 需要提取 sessionId 的正则表达式
3. 断言配置
4. 报告生成命令

# 额外说明
使用环境变量配置 target URL。
```

### 4.3 自动化测试集成（CI/CD）

```markdown
# 角色
测试开发工程师（SDET）。

# 任务
设计一个 GitHub Actions 工作流，用于自动运行测试套件。

# 测试类型
- 单元测试（JUnit）
- 集成测试（Testcontainers：Redis, Milvus）
- 端到端测试（调用真实 LLM 但使用 Mock API 可选）

# 触发条件
- push 到 main 分支
- pull request 到 main
- 手动触发（workflow_dispatch）

# 输出要求
YAML 文件内容，包含：
1. 设置 Java 17 + Maven
2. 启动必要的容器服务（docker-compose 或 service containers）
3. 运行 `mvn test` 和 `mvn verify`
4. 上传测试报告（JUnit XML）到 GitHub Actions artifacts
5. 发送 Slack 通知（可选）

# 注意
避免在 PR 中调用付费 LLM API（使用 mock 或记录重放）。
```

## 六、使用建议

- **选择适合的提示词**：根据当前阶段和角色，复制对应提示词并填写 `{}` 内的占位符。
- **迭代优化**：如果初次输出不符合预期，可以追加要求，例如“请更详细地解释选择理由”或“提供两种备选方案”。
- **输出格式**：提示词中已明确要求格式（表格、代码块、YAML 等），便于直接复制到文档或 IDE。
- **结合工具**：将生成的代码、测试脚本、配置文件直接用于开发环境，生成的文档可导入 Confluence 或 Notion。

通过使用这些结构化的提示词，团队可以更高效地利用 LLM 辅助日常开发工作，确保输出的一致性和专业性。
