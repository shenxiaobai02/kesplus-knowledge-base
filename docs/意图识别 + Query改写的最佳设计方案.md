## 意图识别 + Query改写的最佳设计方案

### 一、方案定位与价值

意图识别与查询改写构成了Agent/RAG系统的**“推理前处理层”**，位于用户输入端与下游执行模块之间。这一层承担着双重核心任务：

- **意图识别**：判断用户“为什么问”（需求类型），将控制权分发给最合适的下游工作流或工具
- **查询改写**：解决用户“怎么问”的问题，将口语化、模糊的原始查询转化为机器可高效检索的精准指令

两者的协同关系可概括为：**意图识别决定策略，查询改写执行策略**。

### 二、核心设计方案概览

当前最成熟可靠的设计方案是一个**分模块解耦的混合架构**，核心组成如下：

| 模块层 | 核心组件 | 实现要点 |
|:---|:---|:---|
| **意图识别层** | 多粒度意图分类器 | BERT/LLM双轨并行，辅以RAG增强分类 |
| **上下文管理** | 对话状态跟踪（DST）+ 短期/长期记忆 | 多轮对话意图延续与槽位填充 |
| **查询改写层** | 规则+LLM双引擎改写 | 同义扩展、指代消解、结构化转换、反事实生成 |
| **路由分发层** | 动态路由决策器 | 基于意图结果分发至RAG检索/数据分析/系统操作等 |

### 三、意图识别模块：精准分类用户需求

#### 3.1 核心分类体系

按分类粒度构建**意图分层体系**：

- **一级意图**（粗粒度）：知识检索、数据分析、系统操作、通用闲聊
- **二级意图**（细粒度）：故障排查、产品咨询、订单状态查询、价格对比等
- **三级意图**（槽位填充）：提取时间、实体、数值约束等细粒度语义元素

**关键设计原则**：保持每个意图分类正交、边界清晰，将Agent能力拆解为最小的功能类别，让模型识别最细粒度的用户需求。

#### 3.2 技术实现方案

意图识别模块采用**“BERT分类 + RAG增强 + LLM兜底”**的三层架构：

| 层级 | 技术方案 | 适用场景 | 优点 |
|:---|:---|:---|:---|
| **第一层：规则快速匹配** | 正则表达式 + 关键词规则 | 高频、结构化明确的查询 | 毫秒级响应，零成本 |
| **第二层：BERT微调分类** | 预训练模型（BERT/RoBERTa）领域微调 | 意图数量5-15个的中频场景 | 准确率高（90%+），延迟低 |
| **第三层：RAG增强分类** | 通过向量检索匹配意图示例，LLM确认 | 长尾查询、泛化要求高的场景 | 泛化性强，无需频繁重训练 |
| **第四层：LLM兜底** | 大模型直接分类 | 复杂、边界模糊的查询 | 利用世界知识，泛化能力最强 |

BERT+RAG的混合方案在实践中能显著提升长尾查询的分类准确率。例如，结合知识图谱做实体泛化和语义矫正，可以有效处理用户多样化表达。

**REIC**（Retrieval-augmented Generation Enhanced Intent Classification）的最新研究进一步验证了RAG增强意图分类的有效性，通过动态检索相关知识，实现高精度分类而无需频繁重训练。

#### 3.3 多轮对话意图追踪

在多轮对话场景中，单轮识别准确率会急剧下降。需配置**对话状态跟踪（DST）模块**：

```
{
  "session_id": "xxx",
  "history": [{"role": "user", "content": "我要退换货"}, {"role": "bot", "content": "请提供订单号"}],
  "current_intent": "return_goods",
  "slots": {"order_id": null}
}
```

该设计可将多轮对话意图识别准确率从40%提升至89%以上。

### 四、查询改写模块：将“人话”翻译为“机器指令”

#### 4.1 核心改写策略

根据不同业务场景，查询改写策略可分为四类：

| 策略类型 | 说明 | 示例 |
|:---|:---|:---|
| **同义扩展** | 将口语词扩展为规范术语，“本子”→“笔记本电脑 Laptop 办公电脑 价格 型号” |
| **指代消解** | 将代词替换为完整表达，“那上一个方法呢”→“上一条回复中提到的第一步骤” |
| **结构化转换** | 将自然语言转为JSON/SQL/Cypher，便于下游精准检索 | 
| **意图补全** | 基于对话历史补充缺失上下文 |

#### 4.2 技术实现方案

**方案对比与选型**：

| 方案 | 实现方式 | 优点 | 缺点 | 适用场景 |
|:---|:---|:---|:---|:---|
| **规则引擎改写** | 正则+同义词词典 | 低延迟、可解释、稳定性高 | 覆盖率有限（通常<40%），无法处理语义变体 | 高频标准化查询 |
| **小模型改写** | T5/BART等轻量模型微调 | 平衡成本与效果 | 训练数据依赖，泛化能力有限 | 中等规模场景 |
| **LLM提示词改写** | 直接调用大模型 | 理解力强，灵活适应各种语义变体 | 成本高、延迟相对高 | 复杂查询、兜底场景 |
| **多智能体改写（前沿）** | 多Agent模拟+进化算法优化 | 最优改写效果，21%+质量提升 | 实现复杂，资源消耗大 | 高价值业务场景 |

在实际系统中，推荐采用**“规则层 → 小模型层 → LLM层”的三级fallback机制**：优先低延迟规则匹配，其次调用小模型处理常见语义变体，最后用LLM兜底处理复杂、边界模糊的查询。

#### 4.3 提示词工程最佳实践

查询改写的Prompt设计需遵循**“任务明确、约束清晰、输入输出结构化”**原则：

```
# Role
你是一个专业的Query改写专家，任务是将用户原始查询转换为对检索器友好的精准指令。

# Rewriting Rules（改写规则）
1. 将口语词替换为规范术语（如“咋样”→“状态如何”）
2. 将指代词补全（如“它”→“上一句提到的A产品”）
3. 将隐含条件显式化（如“便宜的”→“价格低于500元”）
4. 保持原始意图不变，不添加无关信息

# Output Format
{"rewritten_query": "改写后文本", "confidence": 0.95}
```

该模板在企业级Agent系统中已得到验证，可直接复用。

### 五、意图识别与Query改写的协同设计

在实际Agent系统中，意图识别与查询改写必须紧密协同，形成完整的处理闭环：

```
用户输入
    ↓
[意图识别] → 确定意图类别（Knowledge_Retrieval / Data_Analysis / System_Action）
    ↓
[策略选择] → 根据意图类别选择对应改写策略
    ↓
[Query改写] → 调用对应改写引擎，生成优化查询
    ↓
[路由分发] → 根据意图+改写结果分发至下游模块
```

**Key Design Points**：

1. **意图驱动的改写策略**：Knowledge_Retrieval意图 → 聚焦关键词扩展与术语规范化；Data_Analysis意图 → 提取时间、维度约束，转为SQL风格查询

2. **改写结果反馈**：改写后的查询置信度低于阈值时，触发兜底机制或与用户交互澄清

3. **可观测性**：在意图识别和改写节点埋点，记录“原始Query → 意图结果 → 改写后Query → 下游响应”，便于在线评估和迭代优化

### 七、总结与选型建议

| 场景特征 | 推荐方案组合 | 预期效果 |
|:---|:---|:---|
| **意图数量 ≤ 5，业务简单** | 纯LLM Prompt分类 + 基础规则改写 | 开发成本低，迭代快 |
| **意图数量 5-15，B端商业应用** | BERT微调分类 + 规则+LLM改写双轨 | 成本可控，准确率92%+ |
| **意图多样、用户表达多变** | RAG增强分类 + 多策略改写引擎 | 泛化性强，长尾覆盖好 |
| **高价值场景（金融/医疗）** | LLM + 多智能体优化改写 | 最优质量，21%+提升 |

**推荐整体架构**：

- **意图识别**：BERT微调分类（二级意图≈15类）+ RAG增强分类兜底长尾查询
- **查询改写**：规则引擎快速处理高频匹配 + LLM改写处理复杂语义变体
- **协同机制**：意图结果作为改写策略选择的依据，改写后查询路由到对应下游

这一方案已在多个企业级系统中得到验证：某IT支持Agent通过三层核心操作（意图识别+查询重写+路由分发）将用户问题精准导向知识库；在RAG系统中，引入精准意图识别后检索准确率从62%提升至89%。
